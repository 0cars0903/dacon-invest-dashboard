# 데이터감지규칙 (Data Detection Rules)

> 업로드된 투자 데이터(CSV/JSON)의 구조를 자동 분석하여 데이터 유형을 판별한다.
> 이 규칙의 출력은 이후 모든 모듈(지표계산, 시각화매핑, 인사이트생성)의 입력이 된다.

---

## 1. 감지 프로세스

```
파일 업로드
  │
  ├─ 0단계: 전처리 (BOM 제거, 인코딩 감지, 구분자 추론)
  ├─ 1단계: 파일 포맷 파싱 (CSV → Papaparse, JSON → native)
  ├─ 1.5단계: 값 정규화 (콤마·화폐기호·백분율 제거, 타입 캐스팅)
  ├─ 2단계: 컬럼 메타 추출 (이름, 타입, 유니크 수, null 비율)
  ├─ 3단계: 패턴 매칭 (컬럼명 → 데이터 유형 후보)
  ├─ 4단계: 값 분포 검증 (후보 확정)
  └─ 5단계: 데이터 유형 + 서브타입 결정
```

### 0단계: 파일 전처리 (Pre-processing)

파싱 전에 반드시 수행:

| 처리 | 규칙 | 예시 |
|------|------|------|
| BOM 제거 | 파일 시작 `\uFEFF` 감지 시 제거 | `\uFEFF날짜` → `날짜` |
| 인코딩 감지 | UTF-8 우선, 실패 시 EUC-KR → CP949 순서 시도 | 한글 깨짐 방지 |
| 구분자 추론 | 첫 5행 분석: `,` > `\t` > `;` > `\|` 빈도 순 | TSV, 유럽식 CSV 대응 |
| 따옴표 처리 | 값 내 구분자 포함 시 따옴표 내부는 무시 | `"Samsung, Inc."` |

### 1.5단계: 값 정규화 (Value Normalization)

수치 컬럼으로 추정되는 값에서 노이즈를 제거:

```
정규화 파이프라인:
  1. 문자열 trim (앞뒤 공백 제거)
  2. 화폐 기호 제거: /[$₩€¥£]/ → ""
  3. 천단위 콤마 제거: /(\d),(\d)/ → "$1$2"
  4. 백분율 변환: "12.3%" → 0.123 (값 / 100)
  5. 괄호 음수 변환: "(5.2)" → -5.2
  6. 빈 문자열 / "-" / "N/A" / "null" → null 처리
```

| 원본 값 | 정규화 결과 | 비고 |
|---------|-----------|------|
| `"1,234,567"` | `1234567` | 천단위 콤마 제거 |
| `"$50.25"` | `50.25` | 달러 기호 제거 |
| `"₩12,340"` | `12340` | 원화 기호 + 콤마 |
| `"12.3%"` | `0.123` | 백분율 → 소수 |
| `"(5.2)"` | `-5.2` | 회계식 음수 |
| `"-"` | `null` | 결측 처리 |

## 2. 데이터 유형 분류 체계

| 유형 코드 | 유형명 | 설명 |
|-----------|--------|------|
| `STOCK_TS` | 주가 시계열 | 개별 종목 또는 다수 종목의 일봉/주봉 데이터 |
| `ETF_COMP` | ETF 비교 | 복수 ETF/펀드의 가격·수익률 비교 데이터 |
| `PORT_ALLOC` | 포트폴리오 배분 | 자산별 비중·수익률·카테고리 구성 데이터 |
| `CUSTOM` | 커스텀 | 위 유형에 매칭되지 않는 범용 투자 데이터 |

## 3. 컬럼 패턴 매칭 규칙

### 3-1. 날짜 컬럼 감지

다음 패턴 중 하나와 일치하면 **날짜 컬럼**으로 판정:

| 우선순위 | 컬럼명 패턴 (대소문자 무시) | 예시 |
|----------|---------------------------|------|
| 1 | `/^date$/i` | Date, date, DATE |
| 2 | `/^(trade_?date|trading_?date)$/i` | trade_date, TradingDate |
| 3 | `/^(timestamp|time|datetime)$/i` | timestamp, DateTime |
| 4 | `/^(날짜|일자|거래일)$/` | 날짜, 거래일 |

**값 검증**: 아래 형식 순서대로 파싱 시도:

| 우선순위 | 포맷 | 예시 |
|----------|------|------|
| 1 | `YYYY-MM-DD` | 2024-04-12 |
| 2 | `MM/DD/YYYY` | 04/12/2024 |
| 3 | `YYYYMMDD` (8자리 숫자) | 20240412 |
| 4 | `YYYY년 M월 D일` (한국식) | 2024년 4월 12일 |
| 5 | `YYYY.MM.DD` | 2024.04.12 |
| 6 | `DD-Mon-YYYY` | 12-Apr-2024 |
| 7 | Unix timestamp (초/밀리초) | 1712880000 |

모든 포맷 실패 시 해당 컬럼을 날짜에서 제외

### 3-2. 가격 컬럼 감지

| 우선순위 | 컬럼명 패턴 | 매핑 |
|----------|-------------|------|
| 1 | `/^(open|high|low|close|adj_?close)$/i` | OHLC 가격 |
| 2 | `/^(price|value|nav|종가|시가|고가|저가)$/i` | 단일 가격 |
| 3 | `/^(bid|ask|mid)$/i` | 호가 데이터 |

**값 검증**: 양수 실수, 일반적으로 0.01 이상

### 3-3. 거래량 컬럼 감지

| 컬럼명 패턴 | 예시 |
|-------------|------|
| `/^(volume|vol|거래량|거래대금)$/i` | Volume, vol |

**값 검증**: 양의 정수, 일반적으로 100 이상

### 3-4. 수익률 컬럼 감지

| 컬럼명 패턴 | 예시 |
|-------------|------|
| `/^(return|returns|수익률|daily_return|pct_change)$/i` | return, daily_return |

**값 검증**: -1 ~ +∞ 범위, 대부분 -0.3 ~ +0.3 사이

### 3-5. 종목/자산 식별 컬럼 감지

| 컬럼명 패턴 | 예시 |
|-------------|------|
| `/^(ticker|symbol|code|종목코드|종목명|name|asset)$/i` | Ticker, symbol |

**값 검증**: 유니크 값 수가 2 이상, 문자열 타입

**특수 규칙 — 한국 종목코드 처리**:
- 6자리 순숫자 (예: `005930`, `035720`) → 종목코드로 판정, 문자열 강제 변환
- Leading zero 보존: 숫자형 `5930` → 앞자리 0 패딩하여 `"005930"`으로 복원
- 혼합 데이터 (AAPL + 005930) → 전체를 문자열 종목코드로 통합 처리

### 3-6. 비중 컬럼 감지

| 컬럼명 패턴 | 예시 |
|-------------|------|
| `/^(weight|비중|allocation|proportion)$/i` | weight, 비중 |

**값 검증**: 0~1 범위 또는 0~100 범위, 합계 ≈ 1.0 또는 100

### 3-7. 카테고리 컬럼 감지

| 컬럼명 패턴 | 예시 |
|-------------|------|
| `/^(category|sector|industry|분류|섹터|자산유형|asset_?class)$/i` | sector, 자산유형 |

**값 검증**: 유니크 값 수 ≤ 20, 문자열 타입

### 3-8. 통화 컬럼 감지

| 컬럼명 패턴 | 예시 |
|-------------|------|
| `/^(currency|통화|ccy|base_?currency)$/i` | currency, 통화, CCY |

**값 검증**: ISO 4217 코드(KRW, USD, EUR, JPY 등) 또는 통화 기호($, ₩, €, ¥, £)
- 감지 시 `DetectionResult.currencies` 필드에 유니크 통화 목록 기록
- 통화가 2종 이상이면 → 다중 통화 데이터로 판정, 지표계산 시 환산 경고 표시

### 3-9. 평가등급 컬럼 감지 (시멘틱 정규화)

| 컬럼명 패턴 | 예시 |
|-------------|------|
| `/^(rating|grade|opinion|recommendation|투자의견|등급|평가)$/i` | rating, 투자의견 |

**값 검증**: 아래 시멘틱 매핑 테이블로 정규화 (5점 척도):

| 원본 값 | 정규화 점수 | 그룹 |
|---------|-----------|------|
| Strong Buy, 강력매수 | 5 | 매수 |
| Buy, 매수, Outperform, 비중확대 | 4 | 매수 |
| Hold, 보유, Neutral, 중립, Market Perform | 3 | 중립 |
| Underperform, 비중축소 | 2 | 매도 |
| Sell, 매도, Strong Sell | 1 | 매도 |

- 매핑 실패 시 해당 값을 문자열 카테고리로 유지, 경고 표시
- 정규화된 점수는 `columns[].normalizedValues`에 저장

### 3-10. 수정주가 컬럼 감지

| 컬럼명 패턴 | 예시 |
|-------------|------|
| `/^(adj_?close|adjusted_?close|adjusted_?price|수정종가|수정주가)$/i` | Adj Close, 수정종가 |

**값 검증**: 양수 실수, Close 컬럼과 동시 존재 가능
- 수정주가 컬럼 존재 시 → 지표계산에서 Close 대신 수정주가 우선 사용 (`useAdjusted: true`)
- 이를 통해 기업행동(배당·분할) 보정이 반영된 수익률 계산 보장

## 4. 유형 판정 로직

감지된 컬럼 조합으로 데이터 유형을 결정한다.

### 판정 우선순위

```
IF 날짜 컬럼 존재 AND (OHLC 중 2개 이상 존재 OR 가격 컬럼 존재):
    IF 종목 식별 컬럼 존재 AND 유니크 종목 수 ≥ 2:
        → ETF_COMP (복수 종목 비교)
    ELSE:
        → STOCK_TS (단일/소수 종목 시계열)

ELSE IF 비중 컬럼 존재 AND (자산/종목 식별 컬럼 존재):
    → PORT_ALLOC (포트폴리오 배분)

ELSE IF 날짜 컬럼 존재 AND 수익률 컬럼 존재:
    IF 종목 식별 컬럼 존재 AND 유니크 종목 수 ≥ 2:
        → ETF_COMP
    ELSE:
        → STOCK_TS

ELSE:
    → CUSTOM (범용 처리) → 서브타입 판정 진행
```

### CUSTOM 유형의 서브타입 판정 (필수)

CUSTOM으로 분류된 경우 **반드시** 서브타입을 결정한다:

```
IF CUSTOM:
    IF 날짜 컬럼 존재 AND 수치 컬럼 ≥ 1:
        → subType = CUSTOM_TS
    ELSE IF 카테고리 컬럼 존재 AND 수치 컬럼 ≥ 1:
        → subType = CUSTOM_CAT
    ELSE IF 수치 컬럼 ≥ 2:
        → subType = CUSTOM_NUM
    ELSE:
        → subType = CUSTOM_RAW
```

| 조건 | 서브타입 | 설명 |
|------|---------|------|
| 날짜 컬럼 + 수치 컬럼 1개 이상 | `CUSTOM_TS` | 시계열 데이터로 처리 |
| 카테고리 컬럼 + 수치 컬럼 | `CUSTOM_CAT` | 범주별 비교로 처리 |
| 수치 컬럼만 2개 이상 | `CUSTOM_NUM` | 상관관계 분석으로 처리 |
| 그 외 | `CUSTOM_RAW` | 테이블 뷰만 제공 |

## 5. 감지 결과 출력 스키마

```typescript
interface DetectionResult {
  fileType: 'csv' | 'json';
  rowCount: number;
  columnCount: number;
  dataType: 'STOCK_TS' | 'ETF_COMP' | 'PORT_ALLOC' | 'CUSTOM';
  subType: 'CUSTOM_TS' | 'CUSTOM_CAT' | 'CUSTOM_NUM' | 'CUSTOM_RAW' | null;
  // dataType이 CUSTOM이면 subType은 반드시 non-null
  columns: {
    name: string;
    role: 'date' | 'price' | 'ohlc' | 'volume' | 'return'
          | 'ticker' | 'weight' | 'category' | 'currency'
          | 'rating' | 'adjusted_price' | 'numeric' | 'unknown';
    dataType: 'string' | 'number' | 'date';
    nullRate: number;        // 0~1
    uniqueCount: number;
    sampleValues: any[];     // 처음 5개 값
    normalizedValues?: number[];  // 평가등급 시멘틱 정규화 결과 (5점 척도)
  }[];
  tickers?: string[];        // 감지된 종목 목록
  currencies?: string[];     // 감지된 통화 목록 (ISO 4217)
  useAdjusted: boolean;      // 수정주가 컬럼 존재 시 true → 지표계산에서 수정주가 우선
  logScaleRecommended: boolean; // 가격 범위 비율 > 10× 시 true → 시각화에서 로그 스케일 권장
  dateRange?: {
    start: string;           // YYYY-MM-DD
    end: string;
    frequency: 'minute' | 'hourly' | 'daily' | 'weekly' | 'monthly' | 'irregular';
    tradingDays?: number;   // 실제 거래일 수 (연환산 계산용)
  };
  confidence: number;        // 판정 신뢰도 0~1
}
```

## 6. 추가 감지 플래그

### 6-1. 로그 스케일 권장 플래그 (logScaleRecommended)

가격 컬럼의 값 범위가 극단적일 때 시각화에서 로그 스케일을 권장:

```
priceRange = max(price) / min(price)
IF priceRange > 10:
    logScaleRecommended = true
ELSE:
    logScaleRecommended = false
```

- 암호화폐, 고성장주 등 가격 변동 폭이 큰 자산에서 활성화
- ETF_COMP에서 종목 간 가격 범위 차이가 10배 이상이어도 활성화
- 시각화매핑규칙에서 이 플래그를 읽어 Y축 스케일 결정에 반영

### 6-2. 수정주가 우선 사용 플래그 (useAdjusted)

```
IF columns에 'adjusted_price' role 존재:
    useAdjusted = true
ELSE:
    useAdjusted = false
```

- true일 때 지표계산규칙의 모든 가격 기반 지표는 수정주가(Adj Close) 사용
- 이를 통해 배당·분할 보정된 시계열의 연속성 보장

## 7. 에러 처리 규칙

| 상황 | 처리 |
|------|------|
| 빈 파일 | "데이터가 비어있습니다" 에러 반환 |
| 컬럼 없음 (헤더 누락) | 첫 행을 헤더로 추정, 경고 표시 |
| 날짜 파싱 실패 > 50% | 해당 컬럼을 날짜에서 제외, 재판정 |
| 행 수 < 5 | "데이터가 부족합니다 (최소 5행)" 경고 |
| 행 수 > 100,000 | 처음 100,000행만 처리, 경고 표시 |
| 모든 패턴 미매칭 | `CUSTOM_RAW`로 분류, 테이블 뷰 제공 |
| 인코딩 감지 실패 | UTF-8 강제 적용, 깨진 문자 `�`로 대체 후 경고 |
| 구분자 추론 실패 | 콤마(`,`)를 기본 구분자로 사용 |
