# AI 기반 웹사이트 UI/UX 기획·구현 가이드
> 바이브코딩 & 하네스 엔지니어링 접근법 종합 — 2026-04-15

---

## 들어가며

AI에게 UI/UX 구현을 맡기면 결과가 기대보다 낮은 경우가 많다. 이유는 대부분 **AI를 잘못 사용하는 방식** 때문이지, AI의 능력이 부족해서가 아니다. 이 문서는 바이브코딩(Vibe Coding)과 하네스 엔지니어링(Harness Engineering) 두 가지 접근법을 통해 AI와 함께 웹 UI/UX를 기획하고 구현하는 실용 전략을 정리한다.

---

## 1. AI 디자인 품질을 높이는 3원칙

AI에게 디자인을 맡길 때 결과물이 구린 이유는 항상 같다. 아래 세 가지를 지키면 결과물이 달라진다.

### 원칙 1 — 레퍼런스를 반드시 제공하라

AI는 "예쁘게 만들어줘"라는 요청만으로는 어떤 느낌인지 알 수 없다. **좋은 레퍼런스 이미지나 URL을 함께 제공**하면 결과물의 품질이 크게 달라진다.

**레퍼런스 수집에 쓸 수 있는 사이트 유형:**
- 디자인 쇼케이스 사이트 (Dribbble, Behance, Awwwards)
- UI 컴포넌트 갤러리 (UI Verse, Uiverse.io)
- 완성된 웹사이트 스크린샷 아카이브
- 스타일별로 분류된 레퍼런스 모음 사이트 (Google Stitch 관련 생태계)

> **실전 팁:** "이 사이트(URL)의 카드 컴포넌트 스타일로 만들어줘"처럼 레퍼런스를 직접 URL로 붙여주는 방식이 가장 효과적이다.

---

### 원칙 2 — CSS 라이브러리를 하나 고정하라

라이브러리를 지정하지 않으면 AI가 매번 다른 스타일의 CSS를 생성해 일관성이 무너진다. **특정 라이브러리 하나만 명시해도 결과물이 크게 개선된다.**

**추천 CSS/UI 라이브러리:**

| 라이브러리 | 특징 | 적합한 상황 |
|-----------|------|------------|
| **Tailwind CSS** | 유틸리티 클래스 기반, AI가 가장 잘 이해하는 라이브러리 | 커스텀 디자인, 빠른 프로토타이핑 |
| **shadcn/ui** | Tailwind 기반 컴포넌트, Radix UI 접근성 내장 | React 프로젝트, 완성도 높은 UI |
| **DaisyUI** | Tailwind 플러그인, 테마 시스템 강력 | 빠른 MVP, 다크모드 필요 시 |

> **실전 팁:** 프롬프트 첫 줄에 "Tailwind CSS와 shadcn/ui를 사용해서"라고 명시하면 AI가 해당 라이브러리의 컨벤션을 따른다.

---

### 원칙 3 — DESIGN.md를 작성해 AI에게 컨텍스트를 제공하라

Google Stitch에서 발표한 개념으로, **프로젝트의 디자인 시스템을 마크다운 파일로 정리해 AI에게 제공**하는 방식이다. 코딩에서 CLAUDE.md를 쓰는 것과 같은 원리다.

**DESIGN.md 기본 구조:**

```markdown
# DESIGN.md

## 브랜드 정체성
- 톤앤매너: 전문적이면서도 친근한
- 키워드: 신뢰, 간결, 현대적

## 색상 팔레트
- Primary: #3B82F6 (Blue-500)
- Secondary: #10B981 (Emerald-500)
- Background: #F9FAFB
- Text: #111827

## 타이포그래피
- Font: Inter (sans-serif)
- Heading: font-bold, tracking-tight
- Body: font-normal, leading-relaxed

## 컴포넌트 스타일 원칙
- 버튼: rounded-lg, shadow-sm, 호버 시 색상 한 톤 어둡게
- 카드: 흰 배경, subtle border, rounded-xl
- 입력 필드: border-gray-300, focus:ring-2

## 레이아웃 원칙
- 최대 너비: max-w-7xl, 가운데 정렬
- 여백: 섹션 간 py-16, 요소 간 gap-6
- 반응형: mobile-first, sm/md/lg 브레이크포인트

## 참고 레퍼런스
- 전체 분위기: [사이트 URL]
- 카드 스타일: [사이트 URL]
```

> **실전 팁:** DESIGN.md는 일회성이 아니다. 프로젝트가 발전하면서 계속 업데이트해야 한다. 초기에 시간을 투자할수록 이후 AI와의 협업이 훨씬 매끄러워진다.

## 2. 바이브코딩(Vibe Coding) 기반 UI/UX 구현 전략

바이브코딩은 **자연어로 AI에게 구현을 지시하고, 결과를 확인·수정하는 반복 사이클**로 소프트웨어를 만드는 방식이다. UI/UX 구현에 특히 강점이 있다.

### 바이브코딩 UI/UX 사이클

```
1. 아이디어 → DESIGN.md 초안 작성
2. AI에게 전체 레이아웃 초안 요청
3. 브라우저에서 확인 → 피드백
4. 구체적인 컴포넌트별 개선 요청
5. 반복
```

### 효과적인 UI 프롬프트 작성법

**나쁜 예:**
```
로그인 페이지 만들어줘.
```

**좋은 예:**
```
Tailwind CSS와 shadcn/ui를 사용해서 로그인 페이지를 만들어줘.

요구사항:
- 디자인 레퍼런스: Linear.app의 로그인 페이지 스타일 (미니멀, 중앙 정렬)
- 이메일 + 패스워드 입력 필드
- "로그인" 버튼 (Primary 색상, full-width)
- "Google로 계속하기" 소셜 로그인 버튼
- 반응형 (모바일 우선)
- DESIGN.md의 색상 팔레트 적용

컴포넌트 구조:
- pages/login.tsx
- components/LoginForm.tsx
```

### 페이지 유형별 AI 프롬프트 체크리스트

| 페이지 유형 | 명시할 요소 |
|-----------|-----------|
| 랜딩 페이지 | Hero 섹션 구조, CTA 버튼 위치, 애니메이션 여부 |
| 대시보드 | 사이드바 레이아웃, 카드 그리드, 데이터 시각화 라이브러리 |
| 폼 페이지 | 유효성 검사 방식, 에러 표시 방법, 제출 후 동작 |
| 리스트/테이블 | 정렬, 필터, 페이지네이션, 빈 상태 처리 |

## 3. 하네스 엔지니어링(Harness Engineering)으로 AI 변동성 제어

바이브코딩의 약점은 AI 결과물의 **변동성(Variance)**이다. 같은 프롬프트로도 매번 다른 결과가 나오며, 프로젝트가 커질수록 일관성이 무너진다. 하네스 엔지니어링은 이 변동성을 제어해 **예측 가능한 고품질 결과물**을 만드는 전략이다.

### 핵심 개념: 하네스(Harness)란?

AI의 자유도를 적절히 제한하는 **구조적 제약 장치**다. 말을 제어하는 마구(馬具)처럼, AI의 창의성은 살리되 방향을 잡아주는 역할을 한다.

### UI/UX에서 하네스를 적용하는 방법

#### 1) 컴포넌트 라이브러리 하네스
AI가 새 컴포넌트를 만들 때마다 기존 디자인 시스템을 참조하도록 강제한다.

```
# 프롬프트에 항상 포함
"기존 Button, Card, Input 컴포넌트의 패턴을 유지해서 새 컴포넌트를 만들어줘.
임의로 새 스타일을 추가하지 말고, DESIGN.md의 가이드라인을 따를 것."
```

#### 2) 파일 구조 하네스
AI가 파일을 어디에 어떻게 만들지 미리 정의한다.

```
프로젝트 구조:
src/
  components/
    ui/          ← shadcn/ui 원본 컴포넌트 (수정 금지)
    common/      ← 프로젝트 공통 컴포넌트
    features/    ← 기능별 컴포넌트
  pages/         ← 페이지 컴포넌트
  styles/        ← 전역 CSS, 테마
```

#### 3) 리뷰 게이트 하네스
AI가 구현한 UI를 다음 기준으로 검토한 후 다음 단계로 진행한다.

```
UI 리뷰 체크리스트:
□ DESIGN.md의 색상 팔레트를 사용했는가?
□ 지정된 CSS 라이브러리(Tailwind/shadcn)만 사용했는가?
□ 모바일 반응형이 깨지지 않는가?
□ 기존 컴포넌트와 스타일이 일관되는가?
□ 접근성(aria 속성, 키보드 탐색)이 고려되었는가?
```

#### 4) 테스트 하네스 (고급)
UI 스냅샷 테스트로 의도치 않은 스타일 변경을 자동 감지한다.

```bash
# 첫 구현 시 스냅샷 생성
npx playwright screenshot --full-page http://localhost:3000 > baseline.png

# AI가 수정할 때마다 비교
npx pixelmatch baseline.png new.png diff.png
```

## 4. 실전 워크플로우: 처음부터 끝까지

### Phase 1 — 기획 (AI 활용)

```
1. AI에게 경쟁 사이트 분석 요청
   "이런 서비스를 만들려고 해. 유사한 서비스 3개의 UI/UX 특징을 분석해줘."

2. 레퍼런스 수집
   - Dribbble, Awwwards에서 10-20개 레퍼런스 스크린샷 수집
   - 마음에 드는 요소별로 분류 (헤더, 카드, 폼 등)

3. DESIGN.md 초안 작성
   "수집한 레퍼런스를 바탕으로 이 프로젝트의 DESIGN.md를 작성해줘."
```

### Phase 2 — 프로토타이핑 (바이브코딩)

```
1. 전체 페이지 구조 초안
   "DESIGN.md를 기반으로 메인 페이지 와이어프레임 수준의 HTML을 만들어줘."

2. 컴포넌트별 정교화
   "Hero 섹션만 집중해서 다듬어줘. 레퍼런스: [URL]"

3. 반응형 적용
   "모바일에서 확인했을 때 깨지는 부분 수정해줘."
```

### Phase 3 — 품질 관리 (하네스 엔지니어링)

```
1. 컴포넌트 문서화
   "현재 구현된 Button, Card, Input 컴포넌트의 props와 사용법을 정리해줘."

2. 일관성 검사
   "전체 코드에서 DESIGN.md의 색상 팔레트와 다른 색상 값이 쓰인 곳을 찾아줘."

3. 리팩토링
   "중복된 스타일 코드를 공통 컴포넌트로 추출해줘."
```

## 5. 흔한 실수와 해결책

| 실수 | 증상 | 해결책 |
|------|------|--------|
| 레퍼런스 없이 요청 | "예쁘게 만들어줘" → 기대와 다른 결과 | 레퍼런스 URL/이미지 항상 첨부 |
| 라이브러리 미지정 | 페이지마다 다른 스타일 | Tailwind + 하나의 컴포넌트 라이브러리 고정 |
| DESIGN.md 부재 | AI가 매번 다른 색상·폰트 사용 | 프로젝트 시작 시 DESIGN.md 작성 |
| 너무 큰 단위로 요청 | AI가 전체를 한번에 바꿔 기존 부분 파괴 | 컴포넌트 단위로 쪼개서 요청 |
| 검토 없이 다음 단계 | 나중에 발견된 버그가 여러 곳에 전파 | 각 단계마다 리뷰 게이트 통과 확인 |
| 컨텍스트 누락 | AI가 프로젝트 전체 구조를 모르고 수정 | 관련 파일을 함께 붙여서 요청 |

## 6. 도구 추천 스택

```
UI 프레임워크:    Next.js 14+ (App Router)
스타일링:        Tailwind CSS + shadcn/ui
아이콘:          Lucide React
애니메이션:      Framer Motion (간단) / CSS transitions (기본)
폼 관리:         React Hook Form + Zod
상태 관리:       Zustand (클라이언트) / TanStack Query (서버)
AI 코딩 도구:    Claude Code (Cowork), Cursor, Windsurf
```

## 마치며

AI 기반 UI/UX 구현의 핵심은 **AI를 잘 제어하는 것**이다.

- **바이브코딩**은 빠른 프로토타이핑과 아이디어 탐색에 강하다.
- **하네스 엔지니어링**은 결과물의 일관성과 품질을 보장한다.
- **DESIGN.md + 레퍼런스 + 라이브러리 고정**이라는 세 가지 원칙이 두 접근법을 연결하는 기반이다.

처음부터 완벽한 결과를 기대하지 말고, 빠르게 프로토타입을 만들고 하네스를 통해 점진적으로 품질을 높여가는 사이클을 반복하라.
